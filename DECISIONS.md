Ключевые архитектурные решения
1. Эмуляция серверной логики с защитой от Race Condition
Контекст: Необходимо было имитировать реальную серверную среду с возможностью конфликтов при параллельных запросах от разных мастеров.
Решение: Создан MockServer с методом transitionToWork(), который:

Имитирует сетевую задержку (800 мс)

При каждом запросе читает актуальные данные из localStorage

Проверяет статус заявки перед изменением

Возвращает 409 Conflict при попытке изменить уже занятую заявку
Альтернативы: Простая локальная мутация данных без проверок — отклонена, так как не позволяла продемонстрировать проблему гонки.

2. Синхронизация между вкладками через Storage Event
Контекст: При работе в нескольких вкладках браузера данные в них рассинхронизируются.
Решение: Использован window.addEventListener('storage'), который:

Отслеживает изменения в localStorage из других вкладок

Автоматически обновляет локальный массив requests

Перерендеривает текущее представление

Показывает уведомление о синхронизации
Альтернативы: Polling (периодические запросы) — отклонен из-за избыточной нагрузки и задержек.

3. Статусная модель заявок
Контекст: Необходимо отслеживать жизненный цикл заявки.
Решение: Реализован конечный автомат с 5 статусами:

new → только создана, ожидает назначения мастера

assigned → назначен мастер, но ещё не взял в работу

in_progress → мастер взял в работу

done → успешно завершена

canceled → отменена диспетчером
Альтернативы: Упрощенная модель (только "открыта/закрыта") — отклонена, так как не отражает реальный бизнес-процесс.

4. Ролевая модель с динамической маршрутизацией
Контекст: Три разных типа пользователей с разным функционалом.
Решение: Реализована система ролей:

Клиент: только создание заявок

Диспетчер: просмотр всех заявок, назначение мастеров, отмена

Мастер: просмотр своих задач, взятие в работу, завершение
Альтернативы: Отдельные страницы для каждой роли — отклонено в пользу единого интерфейса для простоты.

5. Идемпотентность операций изменения статуса
Контекст: Повторные клики на кнопки не должны создавать побочных эффектов.
Решение: Все операции проверяют текущий статус перед выполнением, кнопки не скрываются, но серверная логика отклоняет некорректные переходы.
Альтернативы: Скрытие кнопок после первого клика — отклонено, так как это маскирует проблему гонки, а не решает её.

6. Хранилище на основе localStorage с версионированием
Контекст: Необходимо сохранять данные между сессиями и обеспечить обмен между вкладками.
Решение:

Ключ srv_requests_v2 с версионированием

Полная замена данных при записи

Чтение актуальной версии перед каждой операцией в MockServer
Альтернативы: IndexedDB — избыточно для данного объема данных. SessionStorage — не подходит для кросс-вкладочного обмена.

7. Визуальная обратная связь для конфликтов
Контекст: Пользователь должен понимать, почему его действие не выполнено.
Решение:

Toast-уведомления с понятными сообщениями

Для конфликтов: "[409 Conflict] Заявка уже занята. Текущий статус: ..."

Автоматическое обновление интерфейса после конфликта
Альтернативы: Простое игнорирование клика — отклонено, так как вводит пользователя в заблуждение.
